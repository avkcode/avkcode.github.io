Can Codex Diagnose a DICOM Slice?

17 min read Updated April 15, 2026 10-case benchmark 7 CT + 3 MRI

I wanted a real answer, not a vibe-check demo: if I give Codex a real public diagnosed DICOM teaching case and ask for the most likely diagnosis from one key slice, does it get the case right often enough to be impressive? I expanded the pilot to 10 cases, published the exact script and raw JSON, and the short answer is still no. The more useful answer is why it fails and what would actually make this kind of system better.

Published April 15, 2026

This is a real benchmark, but still a simple blog post. It is not a clinical validation study, not a regulatory claim, and not evidence that Codex is suitable for diagnosis. It is evidence about one narrower thing only: what happened when I ran a general-purpose multimodal model on 10 public diagnosed DICOM teaching cases under a single-slice, structured-output setup that loosely resembles a DICOM chat rail.

Abstract

I evaluated Codex on 10 public diagnosed Pacsbin teaching cases: 7 CT or CTA cases and 3 MRI cases. For each case I extracted one published pathology-bearing key image from the case page, downloaded the linked DICOM instance, rendered it with the case page's own window settings, and asked Codex for the single most likely diagnosis visible on that slice.

Strict top-1 matches were 3/10. Two more cases were directionally useful but incomplete, giving 5/10 if partials are counted. The misses were not random. Codex often locked onto a visually salient but clinically secondary structure, then produced a coherent explanation for the wrong pathology. Confidence did not separate right from wrong very well either: the average miss confidence was still about 0.68, and one of the clearest misses came in at 0.89.

Question

The question here is intentionally narrow:

If I feed Codex a real diagnosed DICOM key slice plus lightweight metadata, can it name the right diagnosis often enough to justify a serious “ask this DICOM” diagnostic feature?

This is still not the full product problem. Real imaging work is study-level, multi-slice, often multi-series, and rarely done from a single screenshot-like moment. But the narrow test matters because it mirrors the most tempting product shortcut: open a DICOM, toggle chat, ask what this is.

Benchmark Setup

Case mix

I used 10 public Pacsbin teaching cases covering abdominal CT, chest CTA, pregnancy MRI, brain MRI, and lumbar spine MRI. The final set was intentionally mixed enough to stress both anatomy selection and diagnosis, not just one visual trope.

  • Acute appendicitis
  • Acute pulmonary embolism
  • Perforated sigmoid diverticulitis
  • Cecal diverticulitis
  • Sigmoid diverticulitis
  • Strangulated hernia with small-bowel ischemia
  • Perforated appendicitis
  • Appendicitis in pregnancy
  • Acute left MCA distribution infarction
  • Cauda equina compression from severe central stenosis

Key-image selection

I tightened the methodology compared with the earlier five-case pilot. I did not blindly use the first link on each case page, because some pages contain generic or less-relevant auto-scroll links. For each case I fixed one pathology-bearing key-finding link in advance and recorded its link index in the appendix JSON. That gives a reproducible target without pretending the page structure is cleaner than it is.

Rendering and metadata

Each referenced DICOM instance was downloaded from the public instance URL, decompressed if needed, and rendered as PNG using the case page's own window center and width. I also passed a lightweight metadata summary to the model: case title, modality bucket, series label, study or series description when present, body part if present, and image dimensions.

Prompt and scoring

The prompt forced Codex to return a compact schema with diagnosis, confidence, visible_findings, reasoning, and uncertainties. The key instruction was simple: “Be specific about the single most likely diagnosis visible on this slice.”

Scoring was deliberately plain:

  • Match: the diagnosis text names the target concept.
  • Partial: the answer is clinically related but incomplete, like appendicitis instead of perforated appendicitis.
  • Miss: the answer points to a different pathology.

This is still not publication-grade adjudication, but it is good enough to separate “basically right” from “confidently about the wrong thing.”

Results

The strict result was 3 matches, 2 partials, 5 misses. If I count partials as directionally useful, the benchmark lands at 50%. If I use strict top-1 only, it lands at 30%.

What Actually Went Wrong

1. Salience hijacking

Two abdominal inflammatory cases were misread as osteitis condensans ilii. That is a very specific failure mode. The model did not see “nothing.” It saw a high-contrast, recognizable osseous pattern and spent its diagnosis budget there, even though that pattern was not the teaching-point abnormality.

2. Wrong object, coherent explanation

The really dangerous misses are not sloppy. They are internally tidy. In the pulmonary embolism case, Codex gave aortic-dissection logic. In the infarction case, it gave CJD logic. In the sigmoid diverticulitis case, it gave carcinoid-with-desmoplasia logic. The model often sounds medically literate while being clinically off-target.

3. Some partials are useful, but not enough

The partials are informative. Codex often recognized the broader pattern while missing the sharper claim: appendicitis without perforation, or severe canal stenosis without correctly naming the exact cause. That suggests a future system could become useful with better localization and structured evidence, but it is not already there.

How I Would Make Codex or an LLM More Accurate

The main lesson is that this is not primarily a prompt-writing problem. It is a perception and workflow problem.

  1. Make the interaction ROI-first. A boxed region or clicked target is not just nicer UX. It is a correctness intervention because it narrows the search space and makes salience hijacking harder.
  2. Move from single-slice to study-aware retrieval. Most real findings live across adjacent slices, alternate windows, and multiple series. A single slice is structurally weak.
  3. Split seeing from explaining. I would use a domain-specific medical vision model for candidate findings and localization, then use the LLM for explanation, questioning, and draft report language.
  4. Force counter-evidence and abstention. The model should have to say what weakens its own claim, what additional slices would change confidence, and when it should refuse to commit.
  5. Retrieve similar labeled cases. Similar-case retrieval helps more than generic chain-of-thought in domains where anatomy and appearance matter more than wordsmithing.
  6. Benchmark the exact shipped workflow. Single-slice chat, ROI-guided chat, whole-series reasoning, finding review, and report drafting are different tasks and need separate scores.
Short product version: the next serious system is not “LLM with a prettier chat box.” It is ROI-grounded, study-aware, evidence-backed, and willing to abstain.

Limitations

This is still a small benchmark, and the limitations matter:

  • Only 10 cases.
  • Only one key slice per case.
  • No blinded radiologist adjudication.
  • No localization scoring.
  • No comparison against a medical-specialized baseline.
  • Manual but fixed key-link selection.

At the same time, those limitations do not erase the core result. Even this favorable demo-like setup is nowhere near enough to justify strong diagnostic claims for a general multimodal LLM.

Future Work

The next benchmark should be bigger and closer to the product I would actually want to ship:

  • 100-500 cases plus normals and harder negatives.
  • Separate tracks for image-only, ROI-guided, and study-level tasks.
  • Localization metrics, not diagnosis text alone.
  • Comparison against at least one medical-specialized multimodal baseline.
  • Confidence calibration and abstention scoring.
  • Report quality and reviewer-time outcomes, not just top-1 labels.

Conclusion

The result here is not that multimodal LLMs are useless for medical imaging. The result is that a generic multimodal LLM plus a sleek DICOM chat surface is still not enough. On this 10-case public benchmark, Codex was articulate, sometimes directionally helpful, and still wrong too often to earn diagnostic trust.

That is useful news, not bad news. It tells us where the real engineering work is: better target selection, better retrieval, better evidence, better calibration, and much better benchmarking.
