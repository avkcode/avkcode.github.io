Do Not Default to a VPN

3 min read Published April 19, 2026 OpenZiti

Internet blocking pushes people toward public VPNs. That is understandable. It is not a strong security model. If you need private access, host it yourself and keep access tied to identities and specific services.

Published April 19, 2026

Start with the short version. If I want narrow web access for a device, an agent request can look like this: 

Add a new OpenZiti rule for web access to YouTube and Telegram only.

That is the level most people actually want to work at. Define the access, apply it, and move on. The agent can handle the controller objects and the role changes behind it.

Why I Would Not Trust a Public VPN

Internet censorship and blocking are real. A public VPN may be the fastest way around them. But it is still a weak trust model. Your traffic, metadata, and access path move onto someone else's infrastructure. You do not control the server, the logs, the keys, or the operating discipline.

If the goal is security, it is better to run the access layer on your own VM or your own infrastructure. That keeps the trust boundary in your hands. The problem is that running a traditional VPN like OpenVPN is not simple. You still have to manage certificates, client profiles, revocation, routing, DNS, and mobile client behavior.

Why OpenZiti Is Narrower

OpenZiti is useful because it is service-first, not network-first. Instead of putting a device onto part of a routed network, you give an identity access to a specific service.

The technical detail that matters is how the policy is split. OpenZiti has client intercept config for what the client can match, host forwarding config for what the hosting side can forward, and dial and bind policies for who can use or host the service. The hosting side can stay outbound-only. That is narrower than the usual VPN model where authentication often turns into general network reachability.

What The Manual Version Looks Like

The short agent request above is hiding a fair amount of work. Doing the same thing manually takes longer and is easy to get wrong in small ways. 

ssh hawking 'bash -s' <<'EOF'
set -euo pipefail

source /root/.ziti/quickstart/ziti/ziti.env >/dev/null 2>&1
/opt/openziti/bin/ziti edge login \
  "${ZITI_CTRL_EDGE_ADVERTISED_ADDRESS}:${ZITI_CTRL_EDGE_ADVERTISED_PORT}" \
  -u "$ZITI_USER" -p "$ZITI_PWD" -y >/dev/null

cat >/tmp/yt-tg-web.intercept.v1.json <<'JSON'
{
  "protocols": ["tcp", "udp"],
  "addresses": [
    "youtube.com",
    "*.youtube.com",
    "youtu.be",
    "*.googlevideo.com",
    "*.ytimg.com",
    "telegram.org",
    "*.telegram.org",
    "t.me",
    "*.t.me",
    "telegram.me",
    "*.telegram.me",
    "web.telegram.org"
  ],
  "portRanges": [
    { "low": 80, "high": 80 },
    { "low": 443, "high": 443 }
  ]
}
JSON

cat >/tmp/yt-tg-web.host.v1.json <<'JSON'
{
  "forwardAddress": true,
  "allowedAddresses": [
    "youtube.com",
    "*.youtube.com",
    "youtu.be",
    "*.googlevideo.com",
    "*.ytimg.com",
    "telegram.org",
    "*.telegram.org",
    "t.me",
    "*.t.me",
    "telegram.me",
    "*.telegram.me",
    "web.telegram.org"
  ],
  "forwardPort": true,
  "allowedPortRanges": [
    { "low": 80, "high": 80 },
    { "low": 443, "high": 443 }
  ],
  "forwardProtocol": true,
  "allowedProtocols": ["tcp", "udp"]
}
JSON

/opt/openziti/bin/ziti edge create config yt-tg-web.intercept.v1 intercept.v1 -f /tmp/yt-tg-web.intercept.v1.json
/opt/openziti/bin/ziti edge create config yt-tg-web.host.v1 host.v1 -f /tmp/yt-tg-web.host.v1.json
/opt/openziti/bin/ziti edge create service yt-tg-web-egress \
  --configs yt-tg-web.intercept.v1,yt-tg-web.host.v1

/opt/openziti/bin/ziti edge create service-policy yt-tg-web-egress-dial Dial \
  --identity-roles '#yt-tg-web' \
  --service-roles '@yt-tg-web-egress'

/opt/openziti/bin/ziti edge create service-policy yt-tg-web-egress-bind Bind \
  --identity-roles '#yt-host-qs' \
  --service-roles '@yt-tg-web-egress'
EOF

ssh hawking 'source /root/.ziti/quickstart/ziti/ziti.env >/dev/null 2>&1; /opt/openziti/bin/ziti edge update identity anton-iphone-qs3 -a yt-tg-web'

That is why the agent flow matters. The result can stay narrow, but the operator does not need to write and maintain every step by hand.

If the job is just “get around blocking,” a public VPN may be convenient. If the job is private access with a better security model, host it yourself and keep access tied to identities and services instead of broad network reach.
